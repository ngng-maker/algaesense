## What this project is

AlgaeSense is an agentic monitoring/control system for an IoT algae (*Arthrospira*/*Spirulina platensis*) cultivation bioreactor: VOC emissions via an Alphasense PID sensor, biomass via a Pi camera (greenness-based), symbolic regression + active-learning experiment design via the upstream `jaxsr` package, and a conversational agent (Hermes Agent + Slack) that can fit models, suggest next experiments, and adjust the LED — with a human confirmation gate before anything touches hardware.

It started from two spec documents under `Resources/` (`jaxsr_augmentation_spec.md`, the authoritative software spec; `spirulina_voc_experimentalist_protocol.md`, the human lab protocol) describing a `jaxsr-calibration` package. The build went well beyond that original spec's scope — see Architecture below for what actually exists now.

## Architecture

Three packages, each with its own `pyproject.toml`, `src/` (src-layout), and `tests/{unit,integration,fixtures}/`:

- **`jaxsr-calibration`** — never touches hardware. Calibration (standard-addition, reference-jar cross-check, camera zero-point), diagnostics (fleet-zero, ambient baseline, swap-pilot, weekly audit), processing (covariate regression, common-mode subtraction, spectral filtering, dual-rate VOC/camera fusion), raw schemas (`logging_/schema.py`: `VOC_RAW_SCHEMA`, `CAMERA_RAW_SCHEMA`, `ExperimentMeta`). Depends on and calls the real upstream `jaxsr` package directly (`jaxsr.SymbolicRegressor`, `jaxsr.ActiveLearner`, `jaxsr.BasisLibrary`, etc.) — `jaxsr_calibration` is its own top-level import root, not merged into `jaxsr.*`'s namespace (upstream has no plugin mechanism for that). CLI: `jaxsr-cal`.
- **`algaesense-edge`** — runs ON the Raspberry Pi. Sensor acquisition (VOC PID via ADS1115 ADC, optional T/RH via BME280, camera via picamera2), actuator control (LED via WS2811 addressable strip, a `ControlProfileActuator` Protocol so a future heater/cooler can plug into the same control-profile engine — see `actuators/actuators.py`), a small FastAPI network API so the brain machine never needs SSH into the Pi for data. Re-validates every actuator setpoint against configured safety bounds independently of whatever the caller sends (`UnsafeSetpointError`) — this is the actual safety boundary, not a suggestion. `AppState.control_actuators`/`active_control_profiles`/`last_applied_setpoint` are keyed by `(reactor_id, actuator_kind)`, generic across actuator kinds even though LED is the only one implemented; `AppState.led_actuators` stays LED-specific for the manual single-setpoint path. `AcquisitionService.tick_control_profiles` drives whichever actuators have an active profile, once per tick, and caches each successfully-applied value into `last_applied_setpoint` — `run_voc_tick` reads that cache to record the LED's actual PAR into each VOC row (`reactor_par_umol_m2_s`). CLI: `algaesense-edge`.
- **`algaesense-agent`** — runs on the "brain" server, never on the Pi. Six MCP servers Hermes Agent (a real, separately-installed conversational agent) connects to as a client over stdio: `mcp_pipeline` (read-only: fit/suggest via jaxsr, plus `discover_led_response_dynamics` — feeds one experiment's real VOC time-series, with the LED's actually-applied PAR as a second state variable, into `jaxsr.discover_dynamics` for SINDy-style ODE discovery; meant for control-profile runs, where PAR genuinely varies within the run, unlike a static setpoint), `mcp_actuators` (propose/apply split — `apply_led_change`, `apply_led_profile_change`, and `stop_led_profile_change` are the only tools with a side effect; a "profile" is a declarative dict, `{"shape": "ramp"|"sinusoid"|"step"|"constant", ...that shape's own params}` — the agent only ever supplies typed parameters, never code, and `algaesense_edge.actuators.control_profiles.evaluate_control_profile` plus the target actuator's own bounds-check are what actually run it, every tick, on the edge service), `dashboard` (on-demand plot MCP tool + a separate live Streamlit app), `labwiki` (Karpathy LLM-wiki-pattern knowledge base: raw sources, index/log/summaries/entities pages, ingest/query/lint), `mcp_calibration` (guided step-by-step wizards for standard-addition/reference-jar/camera-zero calibration — durable file-backed session state per in-progress calibration, real math from `jaxsr_calibration.calibration`/`camera.calibration`, no hardware touched so no propose/apply gate needed), `mcp_diagnostics` (read-only: fleet-zero/ambient-baseline/swap-pilot/weekly-audit over already-collected raw VOC Parquet files, loaded via `raw_readers.load_raw_voc_readings` — a module-level shared loader, since `mcp_pipeline`'s dynamics-discovery tool needs the identical raw-loading capability). `profile/` holds the Hermes system prompt and config templates — not Python code Hermes imports.

Data layout (unchanged from the original spec):
- Raw: `data/raw/experiments/{experiment_id}/{sensor_id|camera_id}={value}/hour=YYYY-MM-DDTHH.parquet`
- Derived features: `data/derived/features/{campaign_id}/{experiment_id}.parquet`
- Labwiki: `data/labwiki/{campaign_id}/{raw/,wiki/}`

## Setup & running

Each package installs independently:
```
pip install -e packages/jaxsr-calibration
pip install -e "packages/algaesense-edge[hardware]"   # [hardware] only installs/imports on the Pi
pip install -e "packages/algaesense-agent[dev]"
```

Tests, **run per package separately** (not combined in one `pytest` invocation — each package defines its own top-level `tests/__init__.py`, and pytest's import system collides across them if both paths are given at once):
```
pytest packages/jaxsr-calibration/tests
pytest packages/algaesense-edge/tests
pytest packages/algaesense-agent/tests
```

`algaesense-edge` and `algaesense-agent` both use `@pytest.mark.hardware` for tests that need real hardware (WS2811 strip, ADS1115, camera) — skipped by default (`addopts = "-m 'not hardware'"`), run for real on the Pi with `pytest -m hardware`.

For the Streamlit live dashboard: `streamlit run packages/algaesense-agent/src/algaesense_agent/dashboard/streamlit_app.py` (a `.claude/launch.json` config named `streamlit-dashboard` exists for `preview_start`).

For Hermes/Slack/Anthropic setup and Pi hardware/SSH setup: see `docs/slack_and_hermes_setup.md` and `docs/hardware_setup.md`.

## Confirmed real hardware (algaesense-edge)

- **Board**: Raspberry Pi 4 (confirmed 2026-07-17) — matters specifically for the LED below.
- **VOC sensor**: Alphasense PID + ISB (analog 0-3.3V output) → **ADS1115 ADC** (Adafruit STEMMA QT), I2C address **0x48**, channel 0 (A0), single-ended.
- **T/RH sensor**: not yet acquired. `Bme280TRHSensorReader` is a real, ready driver for when one is added — until then, `trh_reader` is `None` throughout and `sample_t_c`/`sample_rh_pct` are recorded as null (schema already supports this).
- **LED**: WS2811 addressable RGB strip (ALITOVE), **GPIO18** (BCM), through a 74AHCT125 level shifter (3.3V→5V) + 470ohm resistor, powered from a separate 12V supply with common ground to the Pi. Driven via `adafruit-circuitpython-neopixel`, **pixel_order="BRG"** (confirmed by testing — not the library's GRB default). `num_pixels` has no default; it must match the real strip's actual count. `rpi_ws281x` (underneath the neopixel library) has solid, long-standing support on Pi 4's GPIO/PWM architecture — this would be a real, current concern on a Pi 5 instead (RP1 chip moved GPIO handling, older PWM+DMA libraries don't uniformly support it yet), but doesn't apply here.
- **Camera**: Raspberry Pi Camera Module v1 (OV5647), CSI connection, via `picamera2`.

## Pi-side OS/permission requirements (not just `pip install`) -- see `docs/hardware_setup.md` for the full walkthrough

- **GPIO/PWM access needs root** — `rpi_ws281x` needs direct DMA/PWM hardware access; run `algaesense-edge start`/`scan-i2c` with `sudo`, using the venv's Python explicitly (`sudo .venv/bin/algaesense-edge ...`) since `sudo` resets `PATH`.
- **Onboard audio conflicts with the LED's PWM channel** — both use the same PWM peripheral. Set `dtparam=audio=off` in `/boot/config.txt` (or `/boot/firmware/config.txt` on newer releases) and reboot before testing the LED.
- **`picamera2` needs `sudo apt install -y python3-picamera2`**, not just the `[hardware]` pip extra — it depends on system `libcamera` bindings pip alone won't provide. If the venv can't see it, recreate the venv with `--system-site-packages`.
- **Likely camera format mismatch, not yet fixed in code**: `picamera2`'s `H264Encoder` writes a raw H.264 elementary stream; `process_clip` reads clips via `cv2.VideoCapture`, which may not open that directly depending on the OpenCV build. Manual workaround: `ffmpeg -i clip.h264 -c copy clip.mp4` before calling `process_clip`. If this turns out to happen every time (not an edge case), fix `Picamera2CameraCapture` to write directly into `.mp4` via picamera2's `FfmpegOutput` instead of leaving this as a manual step.

## Coding conventions

- **Comments**: a short, simple docstring for `help()`/introspection (what it does, no technical detail), plus a *separate* triple-quoted string block placed above the corresponding code for developer-facing rationale/non-obvious detail — not `#` comments, and not folded into the docstring. Blank-line spacing between chunks for readability. This is a deliberate, explicit project convention (not the tool's usual "minimal comments" default).
  - **Exception: Streamlit entry-point scripts** (anything run via `streamlit run x.py`, e.g. `dashboard/streamlit_app.py`). Streamlit's "magic commands" feature renders every bare top-level string expression as literal page content — confirmed by testing — including ones inside a function body once called; only the true first-statement docstring is exempt. Use plain `#` comments in those files instead. This does NOT apply to modules merely imported by a Streamlit app, only to the script actually passed to `streamlit run`.
- **No markdown tables in chat responses to this user** — use lists/headers (horizontal scroll on tables is inconvenient for them). This is about chat responses, not file content — tables inside project markdown docs are fine.
- **Plain-English explanation before code** when explaining how a system/codebase works, especially for "explain X to me" style requests — lead with a narrative story (no file names, no code), then go to code/file-level detail only once asked or for "show me the code" style requests.
- Ousterhout "A Philosophy of Software Design" principles apply throughout: deep modules over shallow ones, avoid needless duplication across subpackages, separate general-purpose from special-purpose code, pull complexity downward, define errors out of existence where reasonable, don't merge genuinely distinct ideas into one file/class just to reduce file count.
- Mock/fake hardware or service stand-ins are **not used** in this codebase as of 2026-07-16 (see Development log) — real hardware classes and real in-process apps (e.g. `httpx.ASGITransport` against the actual FastAPI app) are used instead, even in tests. Pure numeric/text test fixtures for hardware-independent logic (e.g. a synthetic video file to test `process_clip`, synthetic Parquet rows to test a fit) are fine and are not "mocks" in the sense that was ruled out — the distinction is stand-in *hardware/service* implementations vs. plain *input data* for pure functions.

## Known gotchas (found the hard way — don't reintroduce)

- **"PAR" values are an approximation, not a calibrated measurement** — `LEDActuator.par_per_full_duty_umol_m2_s` is meant to come from a real quantum/PAR meter, but absent one, it's derived from a phone lux-meter reading times a rough, LED-spectrum-dependent lux-to-PPFD conversion factor (see `docs/hardware_setup.md` §7). An earlier session's docstring incorrectly attributed this constant to the experimentalist protocol's lux-meter step, which is actually just a one-time photoinhibition safety ceiling check (≤15,000 lux), not a PAR calibration — the two are unrelated measurements that happen to both use a lux meter. Every PAR value anywhere in this system (live setpoints, logged control-profile YAML, any future dynamics-discovery output) inherits this approximation's uncertainty until real PAR-meter equipment replaces it — don't treat a discovered light-response equation's coefficients as physically exact.
- **pyarrow Hive-partitioning false positive**: `pyarrow.parquet.read_table(path)` auto-detects Hive-style partitioning from ANY `key=value` segment in a file's path — including this project's own raw-data layout (`sensor_id=PID01/hour=...parquet`) — and invents a phantom partition column that collides with the real same-named data column already in the file (`ArrowTypeError: Unable to merge: Field sensor_id has incompatible types: string vs dictionary<...>`). Fix: read via `polars.read_parquet(path)` instead (no such auto-detection), `.to_arrow().cast(schema)` if you need it back as a pyarrow Table for `concat_tables`. Affects `algaesense_edge/acquisition/writer.py`; watch for it anywhere else that reads files back from this partition layout with raw pyarrow.
- **Streamlit magic commands** — see Coding conventions above.
- **This repo has no git history before 2026-07-16** — it was a plain directory until a safety checkpoint commit was made right before a large, hard-to-reverse deletion pass (removing all mock/fake test doubles). If you're about to do something similarly sweeping and destructive, check `git status`/commit a checkpoint first, same as then.
- **Cross-package `pytest` collision** — see Setup & running above; always run each package's tests as a separate invocation.
- **`reactor_par_umol_m2_s` is `None` for every VOC row logged before this field started being populated** (recorded from `AppState.last_applied_setpoint`, wired up 2026-07-18) — `discover_led_response_dynamics` raises a clear `ValueError` if the whole column is null for a requested selection rather than feeding NaN into `jaxsr`, but if you're debugging "why does this old experiment have no PAR history," this is why: the row exists, the column just predates the fix. Also a one-tick (~1s) lag: `run_voc_tick` runs before `tick_control_profiles` in `cli.py`'s loop, so a VOC row's cached PAR reflects the *previous* tick's applied value, not the one about to be set this tick — negligible against the PID sensor's own physical response time, but worth knowing if a light/VOC correlation looks off by one sample.

- **Confirmed real Hermes Agent architecture via direct web research before writing any MCP server code**, rather than assuming from training knowledge: it's a real pip package (`hermes-agent`), acts as an MCP *client* (not a place to run server code), and has its own Slack gateway + cron scheduler configured via `~/.hermes/config.yaml` and its own CLI. **Lesson**: for any library/framework claim central to an architecture decision, verify via WebSearch/WebFetch against current docs rather than relying on training-data recall, especially when the claim is old (from an earlier planning session) and might have changed or been imprecise.
- **`neopixel.NeoPixel`'s `pixel_order` accepts a plain string directly** (e.g. `"BRG"`) — it is not required to be one of the library's predefined name constants (`RGB`/`GRB`/`BGR`/`RGBW`/`GRBW`), which don't include `BRG` at all. Caught before shipping by checking the library's real source rather than assuming a `getattr(neopixel, "BRG")` lookup would work.

### 2026-07-17

- **Built the guided calibration wizard** (`algaesense-agent`'s new `mcp_calibration` server): step-by-step standard-addition/reference-jar/camera-zero flows, each backed by a durable YAML session file (`mcp_calibration/sessions.py`) rather than in-memory state, since a session can span many minutes across separate tool calls and an in-memory dict would be lost if the MCP server subprocess restarted mid-session. All the actual math/persistence reuses existing, already-tested `jaxsr_calibration` functions (`fit_sensitivity_per_sensor`, `compute_fleet_ratios`, `compute_blank_baseline` + their real `persist_*` functions) — the wizard's own code is only sequencing and data collection.
- **A tool function's `from x import Y` placed *inside* the function body can't be monkeypatched via `monkeypatch.setattr(module, "Y", ...)`** — the patch target has to be a module-level name for that to work, since a function-local import re-resolves `Y` fresh from its own source module on every call, ignoring whatever the test patched onto the *calling* module's namespace. Hit this writing `record_camera_zero_step_from_edge`'s test; fixed by moving the `EdgeClient` import to module level and introducing an overridable `_build_edge_client()` factory function (mirroring the pattern `mcp_actuators/server.py` already used for exactly this reason) as the actual patch target. **Lesson**: any MCP tool that constructs a network client internally should build it via a small local factory function, not inline in the tool body, purely so tests can inject a fake transport through it later.
- **`EdgeClient` only had `recent_voc_readings`, no `recent_camera_readings`**, even though `algaesense-edge`'s API has exposed `GET /sensors/camera/recent` since Phase 1b — the gap only became visible when the camera-zero calibration wizard needed to pull a live camera reading. **Lesson**: when a new feature needs to call an existing service, double-check the client wrapper actually has a method for every endpoint the service exposes, don't assume symmetry with a sibling method (`recent_voc_readings`) that happens to already exist.
- **Confirmed the user's Pi is a Raspberry Pi 4** (not a Pi 5) — removes the `rpi_ws281x`/RP1-chip compatibility concern for the LED strip. Also confirmed this dev laptop has no network path to the Pi even when the user is simultaneously on a Tailscale-connected second laptop and remote-controlling this one from it — remote desktop/screen sharing doesn't route the controlled machine's network traffic through the controller's VPN. **Lesson**: don't assume a network path exists just because the user has one from their own vantage point; verify directly (a plain `ping`) before planning around it.
- **Built `mcp_diagnostics`** (fleet-zero/ambient-baseline/swap-pilot/weekly-audit as MCP tools). One real gotcha: `AmbientBaselineResult.covariate_models` holds `CovariateModel` objects whose `covariance` field is a raw numpy array and whose `symbolic_regressor` field can be a live, unpicklable `jaxsr.SymbolicRegressor` — neither survives a generic `dataclasses.asdict()` + JSON round-trip the way every other diagnostic result here does. **Lesson**: before wrapping any dataclass-returning function as an MCP tool, check every field's actual type for numpy arrays or live library objects, not just assume `asdict()` is JSON-safe because it worked for the last three tools.

### 2026-07-18

- **Built agent-generated LED control profiles end-to-end**, resolving the design fork noted above: user explicitly picked the declarative-parameters option over arbitrary generated code. Shape: `algaesense_edge.actuators.control_profiles.evaluate_control_profile` (pure function, four shapes: constant/ramp/sinusoid/step) is the only code that ever runs a profile; `AppState.active_led_profiles` tracks one `ActiveLedProfile` (profile dict + start time) per reactor; `AcquisitionService.tick_led_profiles` evaluates and applies every reactor's active profile once per acquisition tick (~1 Hz, same cadence as VOC sampling) via the existing `LEDActuator.set_par()` — meaning the bounds-check re-runs continuously for the life of a running profile, not just once at start. A rejected value stops the profile and turns the LED off rather than continuing with a partially-applied schedule. New edge API endpoints (`POST`/`DELETE /actuators/led/{reactor_id}/profile`) start/stop a profile and best-effort log a YAML record of what was started under `data/raw/experiments/{experiment_id}/control_profiles/`; new `mcp_actuators` tools (`propose_led_profile_change`/`apply_led_profile_change`/`stop_led_profile_change`) expose this to Hermes, with `stop_*` deliberately exempted from the propose-first confirmation gate (stopping an already-approved action needs no fresh review).
- **Starting/stopping a profile never touches real hardware — only a running profile's per-tick evaluation does.** This mattered for test design: every test for the new `POST`/`DELETE /actuators/led/{reactor_id}/profile` endpoints and the new `mcp_actuators` tools runs with no `@pytest.mark.hardware` mark at all (they only exercise `AppState`'s in-memory validate-and-record logic), while `AcquisitionService.tick_led_profiles`'s own tests need the mark for both its "applied" branch (calls real `LEDActuator.set_par` → `hardware.set_duty_cycle`) and its "rejected" branch (calls `turn_off()`, which also reaches real hardware) — only the "no active profile" and "no registered actuator" branches are hardware-free. **Lesson**: when a new method has multiple branches, check each one individually for whether it reaches a real hardware/service call before deciding which tests need the `hardware` mark — don't assume the whole method is one hardware/non-hardware unit. (`tick_led_profiles`/`active_led_profiles`/`start_led_profile`/`stop_led_profile` named in this entry were renamed to their generic `*_control_profiles`/`ActiveControlProfile` equivalents later the same day — see the entry below.)
- **Generalized the control-profile engine to be actuator-kind-agnostic, and fixed two real inaccuracies, before building the JAXSR dynamics-discovery integration.** User asked how JAXSR's real capabilities connect to control profiles; while scoping that (a parked sub-plan — see `C:\Users\JenLab\.claude\plans\jazzy-kindling-hickey.md` if it still exists at time of reading), two things surfaced: (1) control profiles were 100% LED-hardcoded even though `TemperatureActuator`/`StirringActuator` already exist as unimplemented stubs — fixed by adding a `ControlProfileActuator` Protocol (`apply_setpoint`/`turn_off`) and re-keying `AppState`'s tracking dicts to `(reactor_id, actuator_kind)`, while deliberately leaving the LED-specific manual-setpoint path and profile API routes/MCP tool names untouched (no functional payoff to renaming those until a second actuator kind exists); (2) `LEDActuator`'s docstring incorrectly attributed `par_per_full_duty_umol_m2_s` to the experimentalist protocol's lux-meter step, which is actually just a one-time photoinhibition safety ceiling check (≤15,000 lux), not a PAR calibration — checking the actual source document (`Resources/spirulina_voc_experimentalist_protocol.md:124`) rather than trusting an earlier session's claim caught this. **Lesson**: (a) when a user's question exposes that two features (control profiles, future actuators) should share an abstraction, generalize the shared engine before building the next feature on top of the narrower version — cheaper than a rewrite later; (b) a docstring's claim about *why* a design decision was made is itself a claim to verify against the actual source, not just trust from an earlier session, especially when it's about to matter for a new feature (PAR values feeding into a "real" scientific ODE-discovery result).
