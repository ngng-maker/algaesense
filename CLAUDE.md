## What this project is

AlgaeSense is an agentic monitoring/control system for an IoT algae (*Arthrospira*/*Spirulina platensis*) cultivation bioreactor: VOC emissions via an Alphasense PID sensor, biomass via a Pi camera (greenness-based), symbolic regression + active-learning experiment design via the upstream `jaxsr` package, and a conversational agent (Hermes Agent + Slack) that can fit models, suggest next experiments, and adjust the LED â€” with a human confirmation gate before anything touches hardware.

It started from two spec documents under `Resources/` (`jaxsr_augmentation_spec.md`, the authoritative software spec; `spirulina_voc_experimentalist_protocol.md`, the human lab protocol) describing a `jaxsr-calibration` package. The build went well beyond that original spec's scope â€” see Architecture below for what actually exists now.

## Architecture

Three packages, each with its own `pyproject.toml`, `src/` (src-layout), and `tests/{unit,integration,fixtures}/`:

- **`jaxsr-calibration`** â€” never touches hardware. Calibration (standard-addition, reference-jar cross-check, camera zero-point), diagnostics (fleet-zero, ambient baseline, swap-pilot, weekly audit), processing (covariate regression, common-mode subtraction, spectral filtering, dual-rate VOC/camera fusion), raw schemas (`logging_/schema.py`: `VOC_RAW_SCHEMA`, `CAMERA_RAW_SCHEMA`, `ExperimentMeta`), pluggable remote storage (`storage/`: `RemoteStorageBackend` Protocol â€” `upload_file`/`download_file`/`list_keys` â€” plus `LocalDiskBackend`, `FirebaseStorageBackend`, and `SftpStorageBackend` implementations and a `get_storage_backend(config)` factory; see `docs/remote_storage_setup.md`). Depends on and calls the real upstream `jaxsr` package directly (`jaxsr.SymbolicRegressor`, `jaxsr.ActiveLearner`, `jaxsr.BasisLibrary`, etc.) â€” `jaxsr_calibration` is its own top-level import root, not merged into `jaxsr.*`'s namespace (upstream has no plugin mechanism for that). CLI: `jaxsr-cal`.
- **`algaesense-edge`** â€” runs ON the Raspberry Pi. Sensor acquisition (VOC PID via ADS1115 ADC, optional T/RH via BME280, camera via picamera2), actuator control (LED via WS2811 addressable strip, a `ControlProfileActuator` Protocol so a future heater/cooler can plug into the same control-profile engine â€” see `actuators/actuators.py`), a small FastAPI network API so the brain machine never needs SSH into the Pi for data. Re-validates every actuator setpoint against configured safety bounds independently of whatever the caller sends (`UnsafeSetpointError`) â€” this is the actual safety boundary, not a suggestion. `AppState.control_actuators`/`active_control_profiles`/`last_applied_setpoint` are keyed by `(reactor_id, actuator_kind)`, generic across actuator kinds even though LED is the only one implemented; `AppState.led_actuators` stays LED-specific for the manual single-setpoint path. `AcquisitionService.tick_control_profiles` drives whichever actuators have an active profile, once per tick, and caches each successfully-applied value into `last_applied_setpoint` â€” `run_voc_tick` reads that cache to record the LED's actual PAR into each VOC row (`reactor_par_umol_m2_s`). `PartitionedParquetWriter` optionally takes a `remote_backend` (from `jaxsr_calibration.storage`, selected via `cli.py`'s `--storage-backend none|local|firebase|sftp`) â€” when set, each hour's file is uploaded then its local copy deleted immediately, and a restart mid-hour with the local copy already gone re-hydrates it from remote before appending, rather than needing a separate remote-merge path. CLI: `algaesense-edge`.
- **`algaesense-agent`** â€” runs on the "brain" server, never on the Pi. Six MCP servers Hermes Agent (a real, separately-installed conversational agent) connects to as a client over stdio: `mcp_pipeline` (read-only: fit/suggest via jaxsr, plus `suggest_next_experiment_conditions_with_context` â€” the same JAXSR active-learning suggestion, plus an automatic `query_labwiki` lookup (via `labwiki.wiki.query_labwiki`) for the campaign and every condition/target the fit used, surfacing relevant past findings (operator notes, prior fits) alongside the new numeric suggestion in one call; returns whole-page `full_content` for each match, not just `query_labwiki`'s own line-filtered `matching_lines`, since a page's most useful content (an operator's note) is often a different line than whatever literally matched the search topic; does NOT feed labwiki content into JAXSR itself, which stays a pure numerical optimizer over the campaign's real data; both `suggest_next_experiment_conditions`/`suggest_next_experiment_conditions_with_context` also accept an optional `bound_overrides` (per-feature `[lo, hi]`) that DOES genuinely change what JAXSR searches over -- intersected with (never widened past) whatever base range results via `_apply_bound_overrides`; a separate optional `search_bounds` parameter REPLACES the default observed-data-derived base range for named features and CAN widen past it (`_resolve_search_bounds`, applied before `bound_overrides`) -- the fix for a real limitation confirmed while building this project's own synthetic benchmark (see 2026-07-22's dev log): without it, a campaign seeded with only a few clustered experiments could never be suggested a point outside that narrow range, no matter how many rounds ran; `search_bounds`/`bound_overrides` compose (search_bounds sets the start, bound_overrides narrows further), and the resolved `search_bounds` field is reported separately from `fit.feature_bounds` so both "what the data covers" and "what was actually searched" stay visible; the intended workflow (per `system_prompt.md`) is Hermes calling the suggestion tool once plain, reasoning over the fit's trends and the labwiki findings' full content for a genuine constraint, then calling it again with `bound_overrides` (and/or `search_bounds`, if the true physical range is known but unobserved) if one is found, showing both results â€” turning a note into a numeric constraint is Hermes's judgment call, not something `jaxsr` does itself â€” plus `discover_led_response_dynamics` â€” feeds one experiment's real VOC time-series, with the LED's actually-applied PAR as a second state variable, into `jaxsr.discover_dynamics` for SINDy-style ODE discovery; meant for control-profile runs, where PAR genuinely varies within the run, unlike a static setpoint; accepts an optional `ambient_baseline_run_id` â€” a prior `run_ambient_baseline_check(..., persist_run_id=...)`'s id â€” to apply that sensor's persisted ambient-covariate correction before calibration, via new `persist_covariate_models`/`load_covariate_models` in `jaxsr_calibration.processing.covariate`; omitted by default for backward compatibility, confirmed in the same synthetic benchmark to meaningfully and consistently improve dynamics recovery when supplied), `mcp_actuators` (propose/apply split â€” `apply_led_change`, `apply_led_profile_change`, and `stop_led_profile_change` are the only tools with a side effect; a "profile" is a declarative dict, `{"shape": "ramp"|"sinusoid"|"step"|"constant", ...that shape's own params}` â€” the agent only ever supplies typed parameters, never code, and `algaesense_edge.actuators.control_profiles.evaluate_control_profile` plus the target actuator's own bounds-check are what actually run it, every tick, on the edge service; also exposes `get_recent_voc_readings`/`get_recent_camera_readings`, read-only snapshots of whatever the edge service has buffered right now â€” not a continuous stream, since there's no MCP mechanism to push updates into a chat message over time; the Streamlit dashboard is the actual live view; `propose_start_new_experiment_run`/`apply_start_new_experiment_run` restart algaesense-edge on the Pi over SSH via `algaesense_agent.pi_ssh.connect_to_pi` (same connection as `--pull-from-pi`, reusing the identical `ALGAESENSE_PI_*` env vars) â€” a real, hardware-adjacent action gated by the same propose/confirm pattern as LED changes, since it stops whatever experiment is currently running; needs a narrowly-scoped `NOPASSWD` sudoers rule on the Pi for exactly `systemctl restart algaesense-edge`, see `docs/remote_experiment_control.md`), `dashboard` (on-demand plot MCP tool + a separate live Streamlit app â€” the Streamlit app itself has two views: "Live" polls the edge instance directly with elapsed-time axes (VOC in seconds, camera in hours, both since that experiment's first reading) and an experiment-info header -- VOC is shown as ppm on a fixed 0-5 axis (a rough placeholder linear conversion pending a real standard-addition calibration; see the dev log), and camera shows raw greenness plus, when a camera calibration_run_id is entered in the sidebar, a blank-corrected biomass signal via `jaxsr_calibration.camera.calibration.apply_biomass_calibration`; "Past experiment" reads from `history_db.py`'s local SQLite archive instead, populated by the separate `algaesense-dashboard-sync` CLI either from a local copy of the Pi's raw Parquet files or (its `--storage-backend` flag) synced directly from a configured remote storage backend, replacing the manual `scp` step â€” this archive is a queryable *cache* built from that Parquet data, not a replacement source of truth for it), `labwiki` (Karpathy LLM-wiki-pattern knowledge base: raw sources, index/log/summaries/entities pages, ingest/query/lint), `mcp_calibration` (guided step-by-step wizards for standard-addition/reference-jar/camera-zero calibration â€” durable file-backed session state per in-progress calibration, real math from `jaxsr_calibration.calibration`/`camera.calibration`, no hardware touched so no propose/apply gate needed), `mcp_diagnostics` (read-only: fleet-zero/ambient-baseline/swap-pilot/weekly-audit over already-collected raw VOC Parquet files, loaded via `raw_readers.load_raw_voc_readings` â€” a module-level shared loader, since `mcp_pipeline`'s dynamics-discovery tool needs the identical raw-loading capability; `run_ambient_baseline_check` accepts an optional `persist_run_id` to save its fitted per-sensor `CovariateModel`s to `data_dir/derived/diagnostics/ambient_baseline/{persist_run_id}`, so `mcp_pipeline`'s `discover_led_response_dynamics` can later apply the same correction via its own `ambient_baseline_run_id` parameter â€” the only side-effect anywhere in this otherwise read-only server, and a narrow, opt-in one). `profile/` holds the Hermes system prompt and config templates â€” not Python code Hermes imports.

Data layout (unchanged from the original spec):
- Raw: `data/raw/experiments/{experiment_id}/{sensor_id|camera_id}={value}/hour=YYYY-MM-DDTHH.parquet`
- Derived features: `data/derived/features/{campaign_id}/{experiment_id}.parquet`
- Labwiki: `data/labwiki/{campaign_id}/{raw/,wiki/}`

## Setup & running

Each package installs independently:
```
pip install -e packages/jaxsr-calibration
pip install -e "packages/algaesense-edge[hardware]"   # [hardware] only installs/imports on the Pi
pip install -e "packages/algaesense-agent[dev]"
```

Tests, **run per package separately** (not combined in one `pytest` invocation â€” each package defines its own top-level `tests/__init__.py`, and pytest's import system collides across them if both paths are given at once):
```
pytest packages/jaxsr-calibration/tests
pytest packages/algaesense-edge/tests
pytest packages/algaesense-agent/tests
```

`algaesense-edge` and `algaesense-agent` both use `@pytest.mark.hardware` for tests that need real hardware (WS2811 strip, ADS1115, camera) â€” skipped by default (`addopts = "-m 'not hardware'"`), run for real on the Pi with `pytest -m hardware`.

For the Streamlit live dashboard: `streamlit run packages/algaesense-agent/src/algaesense_agent/dashboard/streamlit_app.py` (a `.claude/launch.json` config named `streamlit-dashboard` exists for `preview_start`).

For Hermes/Slack/Anthropic setup and Pi hardware/SSH setup: see `docs/slack_and_hermes_setup.md` and `docs/hardware_setup.md`.

## Confirmed real hardware (algaesense-edge)

- **Board**: Raspberry Pi 4 (confirmed 2026-07-17) â€” matters specifically for the LED below.
- **VOC sensor**: Alphasense PID + ISB (analog 0-3.3V output) â†’ **ADS1115 ADC** (Adafruit STEMMA QT), I2C address **0x48**, channel 0 (A0), single-ended.
- **T/RH sensor**: not yet acquired. `Bme280TRHSensorReader` is a real, ready driver for when one is added â€” until then, `trh_reader` is `None` throughout and `sample_t_c`/`sample_rh_pct` are recorded as null (schema already supports this).
- **LED**: WS2811 addressable RGB strip (ALITOVE), **GPIO18** (BCM), through a 74AHCT125 level shifter (3.3Vâ†’5V) + 470ohm resistor, powered from a separate 12V supply with common ground to the Pi. Driven via `adafruit-circuitpython-neopixel`, **pixel_order="BRG"** (confirmed by testing â€” not the library's GRB default). `num_pixels` has no default; it must match the real strip's actual count. `rpi_ws281x` (underneath the neopixel library) has solid, long-standing support on Pi 4's GPIO/PWM architecture â€” this would be a real, current concern on a Pi 5 instead (RP1 chip moved GPIO handling, older PWM+DMA libraries don't uniformly support it yet), but doesn't apply here.
- **Camera**: Raspberry Pi Camera Module v1 (OV5647), CSI connection, via `picamera2`.

## Pi-side OS/permission requirements (not just `pip install`) -- see `docs/hardware_setup.md` for the full walkthrough

- **GPIO/PWM access needs root** â€” `rpi_ws281x` needs direct DMA/PWM hardware access; run `algaesense-edge start`/`scan-i2c` with `sudo`, using the venv's Python explicitly (`sudo .venv/bin/algaesense-edge ...`) since `sudo` resets `PATH`.
- **Onboard audio conflicts with the LED's PWM channel** â€” both use the same PWM peripheral. Set `dtparam=audio=off` in `/boot/config.txt` (or `/boot/firmware/config.txt` on newer releases) and reboot before testing the LED.
- **`picamera2` needs `sudo apt install -y python3-picamera2`**, not just the `[hardware]` pip extra â€” it depends on system `libcamera` bindings pip alone won't provide. If the venv can't see it, recreate the venv with `--system-site-packages`.
- **Camera format mismatch â€” confirmed real, fixed 2026-07-21**: `picamera2`'s `H264Encoder` always writes a raw H.264 elementary stream regardless of the output path's extension; `run_camera_tick` names every clip `.mp4`, so `process_clip`'s `cv2.VideoCapture` reliably failed to open them (`moov atom not found` â€” not an OpenCV-build quirk, the file genuinely wasn't a valid MP4). Fixed: `Picamera2CameraCapture.record_clip` now records through picamera2's `FfmpegOutput`, muxing a real MP4 directly â€” `ffmpeg` is now a required system dependency (`sudo apt install -y ffmpeg`), not just an occasional manual-remux tool.

## Coding conventions

- **Comments**: a short, simple docstring for `help()`/introspection (what it does, no technical detail), plus a *separate* triple-quoted string block placed above the corresponding code for developer-facing rationale/non-obvious detail â€” not `#` comments, and not folded into the docstring. Blank-line spacing between chunks for readability. This is a deliberate, explicit project convention (not the tool's usual "minimal comments" default).
  - **Exception: Streamlit entry-point scripts** (anything run via `streamlit run x.py`, e.g. `dashboard/streamlit_app.py`). Streamlit's "magic commands" feature renders every bare top-level string expression as literal page content â€” confirmed by testing â€” including ones inside a function body once called; only the true first-statement docstring is exempt. Use plain `#` comments in those files instead. This does NOT apply to modules merely imported by a Streamlit app, only to the script actually passed to `streamlit run`.
- **No markdown tables in chat responses to this user** â€” use lists/headers (horizontal scroll on tables is inconvenient for them). This is about chat responses, not file content â€” tables inside project markdown docs are fine.
- **Plain-English explanation before code** when explaining how a system/codebase works, especially for "explain X to me" style requests â€” lead with a narrative story (no file names, no code), then go to code/file-level detail only once asked or for "show me the code" style requests.
- Ousterhout "A Philosophy of Software Design" principles apply throughout: deep modules over shallow ones, avoid needless duplication across subpackages, separate general-purpose from special-purpose code, pull complexity downward, define errors out of existence where reasonable, don't merge genuinely distinct ideas into one file/class just to reduce file count.
- Mock/fake hardware or service stand-ins are **not used** in this codebase as of 2026-07-16 (see Development log) â€” real hardware classes and real in-process apps (e.g. `httpx.ASGITransport` against the actual FastAPI app) are used instead, even in tests. Pure numeric/text test fixtures for hardware-independent logic (e.g. a synthetic video file to test `process_clip`, synthetic Parquet rows to test a fit) are fine and are not "mocks" in the sense that was ruled out â€” the distinction is stand-in *hardware/service* implementations vs. plain *input data* for pure functions.

## Known gotchas (found the hard way â€” don't reintroduce)

- **"PAR" values are an approximation, not a calibrated measurement** â€” `LEDActuator.par_per_full_duty_umol_m2_s` is meant to come from a real quantum/PAR meter, but absent one, it's derived from a phone lux-meter reading times a rough, LED-spectrum-dependent lux-to-PPFD conversion factor (see `docs/hardware_setup.md` Â§7). An earlier session's docstring incorrectly attributed this constant to the experimentalist protocol's lux-meter step, which is actually just a one-time photoinhibition safety ceiling check (â‰¤15,000 lux), not a PAR calibration â€” the two are unrelated measurements that happen to both use a lux meter. Every PAR value anywhere in this system (live setpoints, logged control-profile YAML, any future dynamics-discovery output) inherits this approximation's uncertainty until real PAR-meter equipment replaces it â€” don't treat a discovered light-response equation's coefficients as physically exact.
- **pyarrow Hive-partitioning false positive**: `pyarrow.parquet.read_table(path)` auto-detects Hive-style partitioning from ANY `key=value` segment in a file's path â€” including this project's own raw-data layout (`sensor_id=PID01/hour=...parquet`) â€” and invents a phantom partition column that collides with the real same-named data column already in the file (`ArrowTypeError: Unable to merge: Field sensor_id has incompatible types: string vs dictionary<...>`). Fix: read via `polars.read_parquet(path)` instead (no such auto-detection), `.to_arrow().cast(schema)` if you need it back as a pyarrow Table for `concat_tables`. Affects `algaesense_edge/acquisition/writer.py`; watch for it anywhere else that reads files back from this partition layout with raw pyarrow.
- **Streamlit magic commands** â€” see Coding conventions above.
- **This repo has no git history before 2026-07-16** â€” it was a plain directory until a safety checkpoint commit was made right before a large, hard-to-reverse deletion pass (removing all mock/fake test doubles). If you're about to do something similarly sweeping and destructive, check `git status`/commit a checkpoint first, same as then.
- **Cross-package `pytest` collision** â€” see Setup & running above; always run each package's tests as a separate invocation.
- **`reactor_par_umol_m2_s` is `None` for every VOC row logged before this field started being populated** (recorded from `AppState.last_applied_setpoint`, wired up 2026-07-18) â€” `discover_led_response_dynamics` raises a clear `ValueError` if the whole column is null for a requested selection rather than feeding NaN into `jaxsr`, but if you're debugging "why does this old experiment have no PAR history," this is why: the row exists, the column just predates the fix. Also a one-tick (~1s) lag: `run_voc_tick` runs before `tick_control_profiles` in `cli.py`'s loop, so a VOC row's cached PAR reflects the *previous* tick's applied value, not the one about to be set this tick â€” negligible against the PID sensor's own physical response time, but worth knowing if a light/VOC correlation looks off by one sample.

## Development log

**Standing instruction**: whenever something fails and gets fixed, an approach is confirmed correct after genuine uncertainty, or a non-obvious lesson surfaces during work on this repo, add a dated entry below. Keep entries short: what happened, why, what to do differently (or keep doing) as a result. ### 2026-07-16

- **Removed all mock/fake hardware and service test-doubles project-wide** (`MockLEDHardware`, `MockVOCSensorReader`, `MockTRHSensorReader`, `MockCameraCapture`, `GpiozeroLEDHardware`, `httpx.MockTransport` fake-handler tests, a `FakeSlackChannel` test harness) at the user's explicit request, after they confirmed understanding it would reduce automated test coverage on non-Pi dev machines. Replaced with: real hardware classes (tests split via `@pytest.mark.hardware`), real in-process FastAPI apps via `httpx.ASGITransport` instead of hand-written fake HTTP handlers, and removing the fake-Slack simulation entirely (the propose/apply tool split is what's actually verified, not a simulated chat). **Lesson**: this kind of instruction has large blast radius across the whole codebase â€” always init/checkpoint git first if no version control safety net exists yet, and read every affected file's real, current content before rewriting (don't rely on memory of what was there).
- **`pip install streamlit` silently downgraded `pyarrow` (25.0.0 â†’ 24.0.0)** in the shared venv, which then surfaced the Hive-partitioning bug above as a test failure. **Lesson**: after installing a new dependency into a shared environment, always re-run the full test suite for every package, not just the one being worked on â€” a transitive dependency conflict can break unrelated code silently.
- **Confirmed real Hermes Agent architecture via direct web research before writing any MCP server code**, rather than assuming from training knowledge: it's a real pip package (`hermes-agent`), acts as an MCP *client* (not a place to run server code), and has its own Slack gateway + cron scheduler configured via `~/.hermes/config.yaml` and its own CLI. **Lesson**: for any library/framework claim central to an architecture decision, verify via WebSearch/WebFetch against current docs rather than relying on training-data recall, especially when the claim is old (from an earlier planning session) and might have changed or been imprecise.
- **`neopixel.NeoPixel`'s `pixel_order` accepts a plain string directly** (e.g. `"BRG"`) â€” it is not required to be one of the library's predefined name constants (`RGB`/`GRB`/`BGR`/`RGBW`/`GRBW`), which don't include `BRG` at all. Caught before shipping by checking the library's real source rather than assuming a `getattr(neopixel, "BRG")` lookup would work.

### 2026-07-17

- **Built the guided calibration wizard** (`algaesense-agent`'s new `mcp_calibration` server): step-by-step standard-addition/reference-jar/camera-zero flows, each backed by a durable YAML session file (`mcp_calibration/sessions.py`) rather than in-memory state, since a session can span many minutes across separate tool calls and an in-memory dict would be lost if the MCP server subprocess restarted mid-session. All the actual math/persistence reuses existing, already-tested `jaxsr_calibration` functions (`fit_sensitivity_per_sensor`, `compute_fleet_ratios`, `compute_blank_baseline` + their real `persist_*` functions) â€” the wizard's own code is only sequencing and data collection.
- **A tool function's `from x import Y` placed *inside* the function body can't be monkeypatched via `monkeypatch.setattr(module, "Y", ...)`** â€” the patch target has to be a module-level name for that to work, since a function-local import re-resolves `Y` fresh from its own source module on every call, ignoring whatever the test patched onto the *calling* module's namespace. Hit this writing `record_camera_zero_step_from_edge`'s test; fixed by moving the `EdgeClient` import to module level and introducing an overridable `_build_edge_client()` factory function (mirroring the pattern `mcp_actuators/server.py` already used for exactly this reason) as the actual patch target. **Lesson**: any MCP tool that constructs a network client internally should build it via a small local factory function, not inline in the tool body, purely so tests can inject a fake transport through it later.
- **`EdgeClient` only had `recent_voc_readings`, no `recent_camera_readings`**, even though `algaesense-edge`'s API has exposed `GET /sensors/camera/recent` since Phase 1b â€” the gap only became visible when the camera-zero calibration wizard needed to pull a live camera reading. **Lesson**: when a new feature needs to call an existing service, double-check the client wrapper actually has a method for every endpoint the service exposes, don't assume symmetry with a sibling method (`recent_voc_readings`) that happens to already exist.
